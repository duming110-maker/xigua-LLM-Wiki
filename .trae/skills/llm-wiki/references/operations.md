# 操作手册

---

## 1. init — 初始化知识库

当用户想要初始化知识库（或 `wiki/` 目录不存在）时：

1. 创建目录结构（`wiki/sources/`、`wiki/wiki/`）
2. 写入初始 `index.md`（只有表头，空表格）
3. 写入初始 `log.md`（包含 init 条目）
4. 向用户确认知识库根目录，并建议下一步：「把素材文件放到 `wiki/sources/` 或直接在这里粘贴内容，开始摄入。」

---

## 2. ingest — 摄入新素材

当用户提供来源（URL、文件路径、或粘贴文本）时：

### Step 1：获取原始素材

根据素材来源类型选择采集方式：

- **URL 素材** → Agent 直接调用 `material-collector` skill
- **本地文件 / 粘贴文本** → 调用 `{skill_dir}/scripts/wiki_collector.py`

### Step 2：Agent 分析素材并展示计划

Agent 直接读取素材内容，识别：
- 核心主题和论点
- 新出现的实体（人物、产品、公司等）
- 新出现的概念（技术、框架、理论等）
- 与已有 wiki 页面相关的信息
- **去重检查**：读取 index.md 中已有页面的摘要，判断素材中的知识点是否已在某个 wiki 页面详细展开。如果是，计划中标注"此内容已在 [[页面B]] 展开，页面 A 只建链接"

然后确定每个新页面所属的子目录（`人物/` / `工具与产品/` / `概念/` / `摘要/`），展示确认：

```
素材摘要：
- 标题：xxx
- 来源：xxx
- 内容类型：xxx（视频转录/文章/网页）
- 字数：约 xxx 字

从这份素材中，我识别出以下关键内容：

核心主题：xxx
新概念/实体：
  - xxx（概念 → 概念/）
  - xxx（实体 → 人物/或 工具与产品/）
与已有页面相关的：
  - [[existing-page]] — 关联原因

计划创建/更新以下 wiki 页面：
  - 人物/xxx.md（新建）
  - 概念/yyy.md（更新，新增 xxx 相关内容）

确认这个范围？需要调整吗？
```

用户确认范围后才执行写入。

**注意**：计划只展示一次。后续步骤中不要重复输出已展示过的素材摘要、分析表格或计划内容。

### Step 3：写入/更新 wiki 页面

- **新页面**：根据页面类型确定子目录（人物/工具与产品/概念/摘要），使用 `{skill_dir}/references/page-templates.md` 中的 Prompt 模板创建
- **已有页面**：整合新信息，添加来源；发现矛盾时按 page-templates.md §5 的标记格式处理

一个来源通常涉及 5-15 个 wiki 页面。不要跳过交叉引用。不要静默覆盖矛盾。

### Step 4：更新 index.md

添加新页面到对应分类表格，刷新已变更页面的摘要和更新日期。

**完成准则**：本次新增或变更的每个页面都在 index.md 对应分类表格中有一行，摘要 ≤60 字且与页面顶部摘要一致。

### Step 5：追加 log.md

**追加到文件末尾**（不是插入头部）。log.md 是追加写入模式，最新条目在尾部。

### Step 5.5：标记素材批次已处理（重要，不可跳过）

更新对应素材批次 `_index.md` 的处理状态：
- 将 `- **处理状态**：未处理` 更新为 `- **处理状态**：已处理`

### Step 6：展示变更总结

如果用户确认的计划未修改，只输出一句：「已按计划完成，共新建 N 个页面，更新 M 个页面。」

如果用户确认后有调整，才展示详细变更表格。变更涉及 5 个以上页面时，追加追问：「有需要修改的吗？」

---

## 3. process — 批量处理素材

当用户要求处理已采集但未整合的素材时：

### Step 1：扫描未处理批次

扫描 `wiki/sources/` 下所有批次目录，读取每个批次的 `_index.md`，对比 `wiki/wiki/index.md` 的来源摘要列表，找出未处理批次。

### Step 2：判断并反馈

无未处理素材时，告知用户：「当前没有待处理的素材，所有批次已整合完毕。」

存在未处理素材时，展示未处理素材清单：

```
发现 N 个未处理批次：

1. <YYYY-MM-DD>-<slug>（采集于 YYYY-MM-DD）
   - 素材数：N 个
   - 涉及：xxx

2. <YYYY-MM-DD>-<slug>（采集于 YYYY-MM-DD）
   - 素材数：N 个
   - 涉及：xxx
```

### Step 3：Agent 逐批分析并展示计划

Agent 直接读取每个批次的素材文件，识别核心主题、实体、概念，确定每个页面所属子目录，展示计划：

```
批次「<slug>」提取结果：

核心主题：xxx
新概念/实体：
  - xxx（概念）
  - xxx（实体）
与已有页面相关的：
  - [[existing-page]] — 关联原因

计划创建/更新以下 wiki 页面：
  - 概念/xxx.md（新建）
  - 人物/yyy.md（更新，新增 xxx 相关内容）

确认后开始写入。
```

### Step 4：批量写入

用户确认后，按 ingest 的写入规则批量创建/更新 wiki 页面（子目录 + Prompt 模板）。

### Step 5：更新 index.md 和 log.md

更新 `index.md` 中的页面条目，向 `log.md` **追加到末尾** process 条目。更新对应批次 `_index.md` 的 `**处理状态**` 为 `已处理`。

---

## 4. query — 查询知识库

当用户对知识库提出问题时：

1. 读取 `wiki/wiki/index.md`
2. 定位 2-5 个最相关的页面
3. 读取这些页面（**必须跟随链接**：遇到 `[[页面名]]` 链接时，将链接指向的页面也一并读取，综合所有相关页面内容回答。不能只看链接处的一句话说明就跳过）
4. 综合回答，内联引用：`(→ [[page-name]])`
5. 追加 `log.md`
6. **提议将回答保存为新的 wiki 页面** — 好的分析应该在知识库中积累，就像摄入的素材一样

回答可以采用不同形式：markdown 页面、对比表格、时间线。根据问题选择最合适的形式。

---

## 5. lint — 健康检查

当用户要求知识库健康检查时：

扫描所有页面并报告：
- **矛盾** — 页面间的冲突主张
- **过时信息** — 被更新来源取代的信息（检查 log.md 确认时效性）
- **孤立页面** — 没有其他 wiki 页面链接进来的页面
- **缺失页面** — 在多个页面中被提及但没有自己页面的实体
- **缺失交叉引用** — 应该互相链接但没有的页面
- **索引缺口** — 存在但未在 index.md 中登记的页面

以优先级列表展示发现的问题。

**确认点**：

```
发现以下问题：
1. [高] 2 个页面存在矛盾：xxx vs yyy
2. [中] 1 个孤立页面：xxx（无任何页面链接到它）
3. [低] 3 个缺失交叉引用

要修复哪些？（可以全选、选择部分、或跳过）
```

用户决定修复哪些，执行批准的修复。

修复完成后追加 `log.md`。

---

## 6. digest — 知识库摘要

当用户要求查看知识库概况时：

### Step 1：读取统计信息

读取 `wiki/wiki/index.md`，统计各类型页面数量（实体、概念、来源摘要）。

### Step 2：读取最近动态

读取 `wiki/wiki/log.md`，获取最近 5 条操作记录。

### Step 3：扫描健康指标

扫描所有 wiki 页面，找出：
- **孤立页面** — 无其他页面链接到它
- **缺失交叉引用** — 内容相关但未互相链接的页面对

### Step 4：输出结构化摘要

```
知识库概况
─────────
总页面数：N
  - 实体页面：N
  - 概念页面：N
  - 来源摘要页面：N
最后更新：YYYY-MM-DD

最近动态
─────────
1. [YYYY-MM-DD HH:MM] ingest | 来源标题
2. [YYYY-MM-DD HH:MM] query | 问题摘要
3. ...

知识缺口建议
─────────
- xxx 领域尚无覆盖，建议补充相关素材
- yyy 与 zzz 高度相关但缺少交叉引用
- N 个孤立页面未被引用，考虑整合

健康提示
─────────
- 孤立页面：N 个（xxx.md、yyy.md）
- 缺失交叉引用：N 对
```
