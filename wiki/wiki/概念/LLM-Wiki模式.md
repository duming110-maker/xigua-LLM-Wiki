# LLM Wiki 模式

> LLM 增量构建并维护持久化 wiki，三层架构区别于 RAG 每次从零检索，知识编译后持续累积

## 概述

LLM Wiki 是一种用大语言模型构建个人知识库的模式。与主流 RAG（检索增强生成）不同——RAG 在每次提问时从原始文档中重新检索片段并拼凑答案，知识不累积；LLM Wiki 让 LLM **增量地构建和维护一个持久化的 wiki**：一个位于用户和原始素材之间的、结构化且相互链接的 markdown 文件集合。

当用户加入新素材时，LLM 不是简单地为日后检索做索引，而是阅读素材、提取关键信息、整合进现有 wiki——更新实体页面、修订主题摘要、标注新旧数据矛盾、强化或挑战正在演化的综合判断。知识被**编译一次**，然后**保持最新**，而非每次查询重新推导。

关键区别在于：wiki 是一个**持久的、复利的产物**。交叉引用已经存在，矛盾已被标记，综合判断已反映读过的所有内容。每加入一个素材、每提一个问题，wiki 都变得更丰富。用户负责选材、探索、提问；LLM 负责所有维护工作——总结、交叉引用、归档、记账。

## 核心观点 / 关键事实

- **三层架构**（→ [[LLM-Wiki来源摘要]]）
  - **原始素材层**：策展过的来源文档（文章、论文、图片、数据），不可变，LLM 只读不写，是事实来源
  - **Wiki 层**：LLM 生成的 markdown 文件目录（摘要、实体页、概念页、对比、综述），LLM 完全拥有，负责创建、更新、维护交叉引用、保持一致性
  - **Schema 层**：一份文档（如 CLAUDE.md / AGENTS.md），告知 LLM wiki 的结构、约定和工作流，使 LLM 成为有纪律的维护者而非通用聊天机器人
- **三种操作**（→ [[LLM-Wiki来源摘要]]）
  - **Ingest 摄入**：加入新素材，LLM 阅读→讨论要点→写摘要页→更新 index→更新相关实体/概念页→追加日志；单个素材可能触及 10-15 个 wiki 页面
  - **Query 查询**：提问时 LLM 先读 index 定位相关页面，再深入阅读并综合带引用的回答；好回答可归档为新 wiki 页面
  - **Lint 健康检查**：周期性扫描矛盾、过时主张、孤立页面、缺失页面、缺失交叉引用、索引缺口
- **与 RAG 的本质区别**：RAG 每次从零检索片段拼凑，无累积；LLM Wiki 编译一次后持续维护，交叉引用已存在，矛盾已标记，综合判断已反映所有读过的内容
- **为什么有效**：维护知识库的繁琐部分不是阅读或思考，而是记账——更新交叉引用、保持摘要最新、标注矛盾、维护跨数十页的一致性。人类因维护负担增长快于价值而放弃 wiki；LLM 不会厌倦、不会忘记更新交叉引用、可一次触及 15 个文件
- **规模适用性**：index.md 在中等规模（约 100 个来源、数百个页面）下效果良好，避免嵌入式 RAG 基础设施的必要性
- **可选 CLI 工具**：随 wiki 增长可引入搜索引擎（如 qmd，本地 BM25/向量混合搜索 + LLM 重排序），小规模下 index 文件即足够

## 争议与数据空白

- 文档自述"刻意抽象"，描述的是想法而非具体实现；目录结构、schema 约定、页面格式、工具链均依赖领域、偏好和 LLM 选择
- 未提供与 RAG 系统的定量对比数据（如检索准确率、维护成本、规模上限的实证）
- 规模上限仅给出"中等规模约 100 来源"的经验说法，未界定大规模（数千来源）下的退化点
- 未讨论多用户协作场景下的冲突处理机制
