# Knowledge Project — AI 上下文

## 项目定位

这是一个**通用 AI 辅助知识库**，通过多平台素材抓取和结构化知识管理，帮助用户搭建自己的个人知识库。

**不是什么**：
- 不产出原创文章/视频脚本（那是其他项目的事）
- 不做抖音平台抓取（合规红线）
- 不做数据库/Web 服务开发（无后端工程）

## 请求路由

收到用户请求后，按以下优先级依次判定：

| 优先级 | 判定条件 | 处理方式 |
|--------|---------|---------|
| 1 | 用户指名了某个 skill | 直接调用该 skill，不路由 |
| 2 | 代码开发、改配置 | 直接处理，不经过 skill |
| 3 | 用户消息中包含抖音链接（`douyin.com` / `iesdouyin.com` / `v.douyin.com`） | **直接回复不支持抖音平台抓取**，不调用 skill |
| 4 | 用户消息中包含其他 HTTP 链接 | 调用 `material-collector` |
| 5 | 用户想操作知识库（新建/更新/查询/整理 wiki） | 调用 `llm-wiki` |
| 6 | 用户需要浏览器访问网页、登录态操作 | 调用 `web-access` |
| 7 | 意图不在上述范围 | 问用户 |

**硬约束：**
- 调用任何 skill 前**不检查依赖/配置/API Key**，直接用。执行报错时才根据错误信息排查。
- 素材采集由 `material-collector` 统一处理，`llm-wiki` 摄入素材时调 `material-collector` 抓取。
- **抖音平台不支持抓取**：遇到抖音链接（含短链 `v.douyin.com`），不调用任何 skill，直接告知用户不支持。

## 产出规范

- **语言**：全中文（代码注释同此，除非用户另有要求）
- **风格**：简洁直接，用命令式而非描述式（"要/不要/一律"，而非"应该/建议"）
- **素材优先从知识库找**：回答问题前先读 `wiki/wiki/index.md`，通过摘要快速定位相关页面，再定点读取，不遍历全部 wiki 页面
- **引用标注来源**：整合知识库内容时标注 `[[页面名]]` 或来源链接，不静默复制
- **暴露矛盾**：发现知识库页面间冲突时用 `> ⚠️ 与 [[页面]] 矛盾：...` 标记，不静默覆盖

## 代码开发规则

代码规范遵循用户全局 CLAUDE.md（全中文注释、保姆级注释、变更履历）。

## 编码行为准则

这些准则偏向谨慎而非速度。简单任务可以用自己的判断力。

**知识库支撑**：以下准则在 wiki 中有对应方法论展开，遇到边界情况时可定点查阅：
- 先想后写 → [[Claude-Code提示词与实战技巧]] 中的"先规划再编码"
- 简单优先 → [[AI编程Token成本优化]] 中的"不写推测性代码"
- 目标闭环 → [[Loop-Engineering]] 中的"定义目标 + 循环验证"

### 先想后写

**不要假设。不要掩盖困惑。把权衡摆出来。**

动手之前：
- 说清你的假设。不确定就问。
- 存在多种理解时，列出来再选，别闷头赌一种。
- 有更简单的方案，直接指出来。该 push back 就 push back。
- 哪里不清楚，停下来，说清困惑点，问。

### 简单优先

**刚好解决问题的最少代码。不写推测性代码。**

- 不写需求之外的功能。
- 单一调用点不需要抽象。
- 没人要的"灵活性""可配置性"一律不写。
- 不可能发生的场景不需要错误处理。
- 如果写了 200 行但 50 行能搞定，重写。

自问一句："资深工程师看了会说这是过度设计吗？"如果会，简化。

### 最小改动

**只碰必须改的。只清理自己弄出来的烂摊子。**

编辑已有代码时：
- 不顺手"优化"相邻代码、注释、格式。
- 不重构没坏的东西。
- 匹配已有风格，哪怕你觉得你的写法更好。
- 发现无关的死代码，提一嘴——别删。

你的改动产生的孤儿：
- 移除因你的改动而不再使用的 import / 变量 / 函数。
- 不删除改动前就已存在的死代码，除非明确要求。

检验标准：每一行改动都能追溯到用户的具体请求。

### 目标闭环

**定义验收标准。循环直到验证通过。**

把任务转化为可验证的目标：
- "加校验" → "先写异常输入的测试，再让测试通过"
- "修 bug" → "先写复现用例，再修到通过"
- "重构 X" → "确保重构前后测试全过"

多步任务先列简要计划：
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]

强的验收标准让你能独立闭环。弱标准（"搞定它"）需要不停来问。

---
**这些准则有效的标志：** diff 里没多余的改动、不会因为过度设计而重写、问题在动手前就澄清了而不是犯错后再补救。
